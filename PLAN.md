# 03.KPdash PLAN

## 1. 업무 목적
KP dashboard 관련 데이터 정리, 대시보드 수정, 내보내기, 문서 갱신을 일관되게 운영한다.

## 2. 폴더 역할 맵
- 031.data: 원본/정규화 데이터, 반영 대상 데이터셋
- 032.dashboard: 실제 대시보드 앱/코드
- 033.export: PDF/리포트/스크린샷 등 추출 산출물
- 034.docs: 운영 기준, 지표 정의, 반영 절차 문서

## 3. 기본 흐름
요청 접수 → 데이터 범위 확인 → 데이터 반영 → 대시보드 수정 → 추출 → QA → 공유

## 4. 단계별 처리 기준
### 0. 요청 접수
- 확인할 것: 반영 대상 월/기간, 수정 범위(데이터/레이아웃/문구), 최종 제출 형식.

### 1. 구조 점검
- 입력 데이터는 031.data, 작업 코드는 032.dashboard, 출력은 033.export, 기준 문서는 034.docs를 사용한다.
- 이전 버전 보관 규칙이 필요한지 먼저 판단한다.

### 2. 데이터 확인
- 최신 normalized 데이터가 준비됐는지 본다.
- 지표 정의가 기존 문서와 맞는지 확인한다.
- 판단 기준: 데이터 정의가 불분명하면 대시보드 수정 전에 034.docs 확인/갱신이 선행되어야 한다.

### 3. 반영 설계
- 어떤 데이터 파일을 교체/추가할지, 어떤 화면이 영향받는지 정리한다.
- export 산출물 이름과 저장 경로를 먼저 정한다.

### 4. 실행
- 데이터 반영 후 대시보드 화면을 수정한다.
- 필요한 경우 빌드/실행으로 실제 렌더를 확인한다.

### 5. QA
- 수치, 카드 로직, 표, 차트, 문구, 페이지 레이아웃을 점검한다.
- 스크린샷 기준 QA를 우선한다.
- 판단 기준: 실제 1920x1080 작업영역 이탈 없음, 비율 유지, 주요 수치 일치.

### 6. 추출/전달
- 최종 PDF/이미지/리포트를 033.export에 저장한다.
- 새 기준이 생기면 034.docs에 반영한다.
- 이전 버전은 같은 폴더 bak/로 이동하는지 확인한다.

## 5. 완료 기준
- 데이터 반영 근거가 명확하고
- 화면 QA가 끝났으며
- 최종 export가 033.export에 저장된 상태
